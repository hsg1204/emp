# Advisory Guides

> Advisory references for design thinking and cross-cutting implementation decisions.

---

## Reading Order

Read guides after the shared index and the relevant frontend or backend index.
Use `guides/` only for advisory design support; guides do not define hard constraints.

Common reasons to read guides:

- cross-layer coordination
- reuse and abstraction decisions
- page layout and information hierarchy
- component structure and design consistency

## Usage

Guides are advisory references, not hard implementation constraints.
Use guides to frame questions, compare options, and decide where to look next.
When a guide topic maps to an existing shared, frontend, or backend spec, follow
that spec as the executable source of truth instead of repeating or overriding it.
If a task requirement or a shared, frontend, or backend specification conflicts with a guide, follow the requirement or specification.

**Language**: All documentation should be written in **English**.
